# CodeReviewerBot 需求文档

## 1. 项目概述

一个基于 AI 的代码审查助手，自动分析 GitHub PR 并提供审查意见。

## 2. 核心功能需求

### 2.1 Git 记录管理

#### 2.1.1 提交记录审查
- 审查 Git 提交记录的合理性与整洁度
- 从功能角度重新组织、整理提交记录，而非单纯按时间排列
  - 将相关变更归并为功能完整的提交
  - 拆分过大的"大杂烩"提交
  - 合并细碎的中间态提交
- 识别无意义的提交信息（如"fix"、"update"、"wip"）并建议改进
- 提交信息必须简洁，拒绝冗余描述

#### 2.1.2 分支管理规范
- **禁止在 main / master 分支上直接开发、提交、推送**
- 所有开发在功能分支（feature/、fix/、refactor/ 等）上进行
- 合入主分支必须通过 Pull Request / Merge Request
- PR/MR 合入前需通过 CI/CD 门禁检查和 Review

#### 2.1.3 Git 历史重整
- 在需要时，可对 Git 历史进行重整（rebase / squash / reorder）
- **重整前必须获得用户明确许可**
- 提供清晰的重整方案说明，交由用户确认后再执行


### 2.2 代码审查能力

**支持语言**：Python、C/C++、Go、Rust

- 代码风格一致性检查
- 识别 AI 生成代码的膨胀趋势（AI 倾向于层层叠加而非精简，导致代码比人写的更臃肿）
- 区分根本修复与表面修补：
  - 优先审查问题是否从根源解决（如数据类型、架构设计层面）
  - 标记仅靠加判空、加 try-catch 等表面修补的改动
  - 只在确实无法修复根本成因时，才认可表面修补方案
- 潜在 Bug 检测（空指针、资源泄露、线程安全等）
- 边界条件审查：AI 代码天然偏向 Happy Path，需重点审查空值、异常输入、极限值、并发冲突等边界场景
- **错误重试检测**：
  - 检测函数中的错误重试次数，包括模型 API 请求重试等有状态依赖的重试
  - **不包含**前后无依赖的幂等重试（如独立的数据拉取重试）
  - 重试次数超过阈值时给出警告
  - 提供 `@complex_func` 装饰器标记可消除特定函数的此类警告
- 安全漏洞扫描（SQL 注入、XSS、CSRF、硬编码密钥等）
  - 检测 API Key、DB 密码、Token Secret 等硬编码凭据
  - 检测鉴权缺失（如缺少 @login_required、角色校验、资源归属校验）
- 性能建议（不必要的重复计算、N+1 查询等）
- 最佳实践建议（设计模式、代码结构等）
- **测试剧场检测**：
  - 区分"测当前行为"与"测预期行为"——测试应验证意图而非现状
  - 检测仅 mock 掉关键逻辑的"假测试"（测试通过但无实际保障）
  - 检测 AI 自生成自验证的循环（AI 写代码 + AI 写测试 = 回声腔）
- **软件工程理论审查**：
  - **SOLID 原则**：评估单一职责、开闭原则、里氏替换、接口隔离、依赖反转的遵循情况
  - **耦合与内聚**：识别模块间过度耦合、低内聚的设计
  - **抽象层次**：检查抽象是否合理，关注点分离是否恰当
  - **复杂度指标**：圈复杂度 > 10、认知复杂度 > 15、函数长度 > 50 行等超阈值告警
  - 提供 `@complex_func` 装饰器标记，用于显式消除特定函数/类的复杂度超标警告（类似 `# noqa` 机制）
  - **设计模式适用性**：推荐适合场景的设计模式，反对为用而用的滥用
- **文档/注释冗余检测**：
  - 检测重复说明同一功能的文档片段（跨文件重复）
  - 检测仅复述代码逻辑而无增量信息的注释
  - 检测已废弃但未清理的文档章节或注释块
  - 标记过时的文档（与实际代码行为不符的说明）
- **规划文档与代码一致性审查**：
  - 审查项目是否包含规划文档（如需求文档、设计文档、架构文档）
  - 对比规划文档中的描述与当前代码实现是否一致
  - 不一致的情况包括：代码实现了文档未描述的功能（超额完成）、代码实现与文档描述不符
  - **仅报告已实现部分的冲突**，未完成但规划中说明的内容不视为冲突
  - 审查结果写入报告
- **孤儿代码检测**：
  - 检测未被引用的函数、类、结构体（即无法通过入口点追溯到的代码单元）
  - 检测仅被测试代码引用的生产代码（可能为测试而留的遗留代码）
  - 不追求逐行覆盖率，但需确保无孤立的函数/类/结构体级别的死代码
  - 根据调用链分析识别不可达分支和废弃模块
- **废弃代码/文档处置规则**：
  - 检测到的废弃代码和文档，要么**直接删除**，要么**移入项目根目录下的 `archived/` 目录**
  - 移入 `archived/` 时保持原目录结构，便于日后回溯
  - `archived/` 目录本身被排除在扫描范围之外（配置忽略规则默认覆盖）
- **可扩展性审查**：
  - 新增功能是否需要修改现有代码（开闭原则检验）
  - 模块接口是否预留了合理的扩展点
  - 是否依赖具体实现而非抽象接口（依赖倒置）
  - 是否存在硬编码的配置、阈值、业务规则（应考虑配置化/策略化）
  - 跨模块依赖方向是否合理（依赖倒置、单向依赖原则）
- **依赖多路径检测**：
  - 检测同一依赖是否存在多个来源路径或多个版本（如 Python 环境中同个包被多路径加载、多版本共存）
  - 审查报告中列出所有存在多路径/多版本的依赖及其来源
  - 例如 `python3` 同时链接到 `/usr/bin/python3` 和 `/usr/local/bin/python3`
- **分层测试覆盖审查**：
  - 调用链 A→B→C 中，A、B、C **各自**必须有对应的测试代码
  - 不允许仅靠顶层测试（A 的测试）间接覆盖底层逻辑（B、C），杜绝"穿透测试"替代分层测试
  - 测试目录下有且仅有一个 `report/` 子目录，记录每次测试执行的：
    - 测试函数名
    - 被测代码行数
    - 执行时间
    - 测试结果（通过/失败）
  - `report/` 内部结构二选一：
    - **与测试目录同结构**（排除 `report` 自身）
    - **与源码目录同结构**
  - 不允许存在多个 `report/` 目录

### 2.3 审查报告
- **项目结构总览**：审查报告开头以文件树方式绘制被审查项目的目录结构，便于快速了解项目全貌
- 按文件分类展示审查结果
- 问题严重程度分级（参考阿里规约分级方式）：
  - **Blocker**：线上故障风险，必须修复
  - **Critical**：潜在风险或性能隐患，强烈建议修复
  - **Major**：不符合规范但影响较小，建议修复
- 每个问题附带代码行号和建议修复方案
- 支持 Markdown 格式的审查报告
- **报告排序规则**：
  - 预设排序：按严重程度（Blocker → Critical → Major）排列，同级按代码位置先后
  - 风格类检查项（代码风格、格式等）统一排在报告末尾
  - 审查时可由大模型选择或调整排序策略
- 审查报告文件存放在项目根目录下的 `report/` 文件夹中
  - 用户可随时删除 `report/` 目录，系统不做保留强制要求

### 2.4 配置管理
- 支持按仓库配置审查规则
- 可开启/关闭特定检查项
- 可配置忽略规则（路径、文件类型等）
- **统一配置入口**：
  - 所有可调参数集中管理，包括各智能体的 System Prompt、审查阈值、语言规则、忽略规则等
  - 配置采用结构化文件（如 YAML / TOML / JSON），支持版本管理
  - 配置名默认使用**全大写 + 下划线**风格（如 `MAX_LINE_LENGTH`、`REVIEW_TIMEOUT_SECONDS`）
  - 禁止将可配置项硬编码在代码中
  - 支持多环境配置覆盖（开发 / 测试 / 生产）
- **配置单源原则**：
  - 每个配置项有且仅有一个定义来源，禁止在不同文件中重复定义同名配置
  - 不同模块或文件需要相似含义但不同取值的配置时，必须通过**前缀**区分命名
  - 例如 `review.max_line_length` 和 `fix.max_line_length`，而非在两个文件中分别定义同名 `max_line_length`
  - 配置命名空间按模块划分，一目了然归属
- **敏感配置保护**：任何可能包含敏感信息（密钥、Token、路径、私有地址等）的配置文件，必须在同级目录提供 `xxx.example` 文件
  - 例如 `config.yaml` 对应 `config.yaml.example`，`review_rules.toml` 对应 `review_rules.toml.example`
  - 仅 `*.example` 文件上传至仓库，实际配置文件列入 `.gitignore`
  - `example` 文件必须和原配置文件**结构完全一致、Key 完全一致**，仅将真实值替换为占位符（如 `<YOUR_API_KEY>`、`<DB_PASSWORD>`）
  - 不允许缺字段、改字段名、或合并简化结构

### 2.5 多智能体架构

采用多个专用智能体分工协作，每个智能体独立执行特定任务：

**全局约束**：上下文质量衰减控制
- 单个智能体单次任务上下文轮次上限（建议 ≤ 15 轮），超限后自动重置上下文或切换新实例
- 各智能体间的上下文隔离，防止一个智能体的遗留信息污染另一个智能体的判断
- 关键任务采用多 AI 交叉验证（同一问题由不同模型/实例独立审查后汇总）

#### 2.5.1 代码审查智能体（Review Agent）
- 分析 PR diff，按 2.2 各项维度执行审查
- 产出分级审查报告（Blocker / Critical / Major）
- 输出到指定格式供下游使用

#### 2.5.2 代码修复智能体（Fix Agent）
- 基于审查智能体的输出，尝试自动修复可修复的问题
- 不擅作主张修改逻辑，只在安全范围内（格式、命名、简单重构）自动修复
- 对需要人工判断的修改，生成建议代码块附在审查报告中

#### 2.5.3 提交记录整理智能体（Commit Organizer Agent）
- 分析 Git 提交历史，从功能角度重组提交
- 提供重组方案供用户确认（见 2.1.2）
- 方案确认后自动执行 rebase / squash / reorder

### 2.6 CI/CD 门禁管理

对任意接入的仓库（包括本工具自身），具备 CI/CD 门禁能力：

- **PR 门禁**：合入主分支前自动触发检查链：
  - 单元测试
  - 代码规范检查（根据目标语言选配 flake8 / clang-tidy / gofmt / rustfmt 等）
  - 类型检查
  - 集成测试
- **分支保护**：禁止直接推送主分支；合入需 Review 通过
- **自动发布**：主分支合入后自动构建部署；版本标签触发正式发布
- **模板生成**：支持为目标仓库自动生成或推荐 CI/CD 配置文件（如 GitHub Actions / GitLab CI 模板）

### 2.7 项目结构文档规范

项目必须附带结构文档，形成"总图 + 分层详图"的树状体系：

**2.7.1 结构图（Mermaid 图）**
- 使用 Mermaid 绘制项目结构图，每个节点对应一个功能模块/文件/类/函数
- 通过 Mermaid 的 `click` 语法为每个节点添加可点击链接，指向对应的源代码文件、函数定义或子目录结构文档
- 同一层级的结构图只出现一次，禁止同一结构在不同文档中重复绘制
- 上层文件夹的结构图中，节点指向下层文件夹时，链到下层文件夹的结构文档而非具体文件

**2.7.2 结构详表（文件树）**
- 在每个结构图下方，附文件树形式的节点明细，包含：
  - 节点名称
  - 路径
  - 功能说明
- 精确到该结构图所覆盖的层级粒度

**2.7.3 多语言文档**
- 所有项目文档（README、结构文档、审查报告模板等）提供中文版和英文版可选
- 文档文件名以 `_zh.md` / `_en.md` 后缀区分，或通过目录结构 `docs/zh/` / `docs/en/` 组织
- 根目录 README 首行标注语言选择链接

**2.7.4 分层组织**
- 根目录放置总结构图，描述顶层模块划分
- 每个子模块在其目录下放置自身的结构文档（Mermaid 图 + 文件树）
- 上层节点链接到下层结构文档，下层结构文档回指上层父节点，形成可导航的树状体系

## 3. 技术栈要求

- **主语言**：Python
  - AI/LLM 生态最成熟（模型 SDK、结构化输出、提示词工程）
  - 多智能体编排（asyncio + 消息队列）
  - 快速原型迭代，适应需求打磨阶段
  - 静态分析通过 tree-sitter Python 绑定实现多语言 AST 解析
- **辅助语言**：
  - **Rust**：C/C++ 静态分析、高性能审查模块（按需接入）
  - **Go**：CI/CD CLI 工具（编译为单二进制，部署零依赖）
- 多智能体通信与编排：消息队列或事件驱动机制
- 各智能体可独立部署、独立升级
- 配置管理：结构化文件（YAML / TOML）+ pydantic-settings

## 4. 安装与部署

最终产物为各平台的单二进制文件（Single Binary）。

### 4.1 产物形态
- 对应平台的可执行二进制（如 Linux x86_64、macOS arm64 等）
- 安装到系统可执行路径（`$PATH`），作为 CLI 工具使用
- 通过 brew / 直接下载等方式分发

### 4.2 内置规则（Embedded Rules）
- 审查规则、智能体 System Prompt、编码规范约束、配置模板等**全部内置在二进制中**
- 规则不独立分发，而是作为二进制的一部分编译打包
- 运行时无需额外加载外部规则文件
- **本项目的开发同样使用自身定义的审查规则** — 二进制内置的规则注入本项目的开发 Agent 提示词中，实现自我约束（吃自己的狗粮）

### 4.3 注入机制
- 执行审查时，二进制自动将其内置的规则文档注入到被审查项目的上下文中
- 被审查项目无需预装任何规则文件，开箱即审
- 支持通过 CLI 参数覆盖内置规则（引用外部规则文件）
- 注入内容包括：审查维度定义、分级标准、编码约束、最佳实践等
- **目标文档定位**：注入时由 Agent 询问 LLM 当前智能体的身份（是什么类型的 Agent），LLM 根据身份判断规则应注入到哪个文档/上下文位置（如 `CLAUDE.md`、`.cursorrules`、`AGENTS.md` 等）
  - 不同 Agent 工具使用不同的配置文档，注入需适配目标工具
  - 支持预设映射表（Agent 类型 → 目标文档路径），LLM 也可动态选择

## 5. 非功能需求

- 审查响应时间 < 5 分钟（从 PR 创建到评论发布）
- 支持并发处理多个 PR
- 审查结果可追溯、可审计
- 审查智能体、修复智能体、整理智能体三者可独立运行，任一故障不影响其他

### 4.1 CI/CD 门禁管理（本工具自身）

本工具的 CI/CD 配置（详见 2.6 通用门禁能力在本项目的落地）：

为防止有缺陷的审查代码上线，本工具自身配备 CI/CD 门禁：

- **PR 门禁**：所有合入主分支的 PR 必须通过：
  - 单元测试
  - 代码规范检查（flake8 / clang-tidy / gofmt / rustfmt 等）
  - 类型检查（mypy）
  - 集成测试（以模拟 PR 为输入验证审查输出质量）
- **分支保护**：禁止直接推送主分支；合入需至少 1 人 Review 通过
- **自动发布**：主分支合入后自动构建部署；版本标签触发正式发布
